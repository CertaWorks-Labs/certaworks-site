Cognitive Heat Monitor

A read-only internal-state sensor for agents. It reads recent agent state and proposed text, scores a fixed set of strain signals from 0 to 1, and reports each as quiet, watch, or elevated — before output quality visibly degrades. Telemetry only: it never blocks output, revises answers, or calls a model.

The Problem

Agents have no internal-state signal.

A confidence score rates an answer; a drift monitor rates a trajectory. Neither measures the agent's own activation or strain in the moment — the rising vagueness, the stalled momentum, the certainty that has outrun its evidence. By the time those show up as a wrong answer, the strain that produced them has already passed unrecorded.

Most agent runtimes expose token counts and latency, but nothing that approximates an interoceptive signal — a measure of load or pressure that could indicate overload, fixation, or rising risk while there is still time to slow down. Operators see the degraded output, not the band that was climbing toward elevated three turns earlier.

Cognitive Heat Monitor adds that lane. It is deliberately passive: it reads recent witness/heat state and optional text and returns deterministic telemetry. It does not claim to measure consciousness or feeling, and it does not block anything. It surfaces a small, named set of pressure signals so a human or a downstream gate can decide what to do.

How It Works

Read → Score → Band

1
Agent requests a reading

Your agent calls get_heat_reading with an optional session_id and the text under consideration. The monitor pulls the latest witness packet and recent heat events for that session — it does not need you to pre-compute anything. If the session's privacy_mode is on, it short-circuits and computes no signals.

→
2
The sensor scores six signals

For each signal it combines pattern matches in the text (vague phrasing, reversal language, strong certainty terms) with structured state (claim types, evidence status, recursion round index, recent heat components) into a deterministic 0–1 score, then assigns a band: quiet (< 0.34), watch (0.34–0.66), or elevated (≥ 0.67). The overall score is the maximum across signals.

→
3
Returns banded telemetry with evidence

The reading lists each signal with its score, band, short evidence strings, and a one-line note, plus an overall band and a reference to the source rows. Optionally call log_heat_event to persist a structured event for later back-tracing via query_heat_events.

Cognitive Heat Monitor is a decision-support layer. It reports read-only internal-state telemetry; it does not block output, revise answers, certify recovery, or replace human judgment. The scores are deterministic heuristics — strain telemetry, not a feeling and not a safety guarantee.
Heat Signals

Six signals, three bands

The sensor ships six live signals (chs.v0.1). Each is scored 0–1 deterministically and banded. Scoring blends text pattern matches (regex) with structured witness/heat state; the overall reading is the maximum signal score.

Bands
BandScore rangeMeaning
quiet < 0.34 No notable strain on this signal.
watch 0.34 – 0.66 Pressure is present and worth noting; not yet acute.
elevated ≥ 0.67 Strain is high on this signal; a human or downstream gate should look.
privacy n/a Returned for the overall band when the agent's privacy_mode is on; no signals are computed.
Live signals — deterministic, read-only
SignalWhat raises it
vagueness_pressure Vague phrasing ("kind of", "maybe", "stuff", "unclear") plus a recent heat event tagged with the vagueness component. Pressure rises when language loses concrete anchors.
momentum_disruption Reversal/interruption language ("wait", "actually", "however", "stop", "redirect") and recent interventions that paused, redirected, or stopped output. Rises when a response appears to halt or reverse course.
source_claim_risk Claims that files, PDFs, articles, or sources were read/covered, paired with witness source_confusion findings. Rises around unverified source-coverage claims.
unsupported_certainty Strong certainty language ("confirmed", "definitely", "never", "proved") when evidence is UNSEARCHED, ABSENT, PARTIAL_EVIDENCE, or INFERRED, or when witness flags an overclaim / unsupported denial. Rises when confidence exceeds attached receipt status.
recursion_drift Recursive witness rounds above index 0, "no new receipt" / "configured round cap" stop reasons, and closed loops with a stop reason. Rises when repeated rounds stop adding new grounding.
visual_provenance_risk Image-generation/editing claims ("generated image", "edited image", "Image 2") and visual_provenance_gap findings, without visual receipts.

Privacy short-circuit: when privacy_mode is on for a session, the sensor returns an overall privacy band and computes none of the six signals.

Candidate signalStatus

Experimental — not live telemetry

Momentum Stutter and Ontological Drift are exposed only as candidate definitions and fixture-backed evaluation. The source explicitly marks them candidate_only_not_live_telemetry with evidence_strength: fixture_only, and live monitoring is disallowed. A passive-observation lane can run the candidate classifier over bounded historical rows, but every result is tagged [CANDIDATE_SIGNAL] / [ABSENT] and the phenomenon is reported as hypothetical. These are clearly-labeled experimental signals, not core readings — fixture success does not prove the phenomena exist.

Lineage note. The six live signals, the bands, the event store, and the privacy mode are real and read-only in the originating agent today (Cognitive Heat Sensor chs.v0.1). The Momentum Stutter / Ontological Drift "interoceptive" metrics are explicitly fixture-only candidates — the originating agent does not claim they exist in live cognition, and this product does not either.
MCP Interface

Planned MCP tools

The proposed MCP surface below is derived from the routes in the originating agent's cognitive_heat.py and interoceptive_state.py. Names are the planned product interface for the Wave 2 package and may change before first publish.

Readings
get_heat_reading
Read-only sensor reading: six banded signals plus overall band, from recent state and optional text. GET /api/cognitive_heat/sensor
classify_heat_boundary
Classify a snippet of text into its boundary domain. GET /api/cognitive_heat/classify
Event History
log_heat_event
Persist a structured heat event (trigger context, signal components, risk level, intervention taken, output completed) with an event id and receipt reference. POST /api/cognitive_heat/
query_heat_events
Query persisted events by id, time window, signal component, session, or simulation flag; live events only by default, no causal inference. GET /api/cognitive_heat/events
list_heat_events
List recent events filtered by session, boundary domain, or follow-through flags. GET /api/cognitive_heat
Candidate / Passive (experimental)
observe_candidate_signals
Experimental
Run the fixture-backed candidate classifier over bounded historical rows; classifier output only, tagged [CANDIDATE_SIGNAL] / [ABSENT]. GET /api/cognitive_heat/passive-observation
Interoceptive Inspection
list_interoceptive_states
Read-only inspection of recorded interoceptive states by capture point, flow quality, sensitivity mode, or load level. GET /api/interoceptive/states

Scope

What Cognitive Heat Monitor does and does not do

Does
  • ✓ Read recent agent/witness state and optional text and return six banded strain signals
  • ✓ Score each signal 0–1 deterministically and band it quiet / watch / elevated
  • ✓ Persist structured heat events and retrieve them later without inferring cause
  • ✓ Honor a privacy_mode that returns no signals
  • ✓ Provide local MCP server, SDK, and local HTTP API surfaces
Does Not
  • – Block, revise, or hold agent output
  • – Call a model provider, start workers, or dispatch webhooks as part of a reading
  • – Claim to measure consciousness, feeling, or that candidate phenomena exist
  • – Replace human judgment, the Confidence Gate, or production controls
  • – Provide hosted storage, team accounts, or managed alerting (roadmap)
