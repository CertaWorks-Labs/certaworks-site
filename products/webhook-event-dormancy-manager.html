Webhook & Event Dormancy Manager

Dormant by default — external calls fire only when you say so. Agents queue webhooks and emit events without ever touching the outside world by accident. Delivery is fail-closed: an event can be queued and inspected, but nothing leaves the machine until an explicit, approved activation branch is opened.

The Problem

An unintended external call is one stray code path away.

An agent that can emit webhooks or fire external events is one stray code path away from an unintended side effect — a message sent to a real Slack channel, a callback POSTed to a production endpoint, a retry storm replayed against a third party. The danger is rarely a deliberate decision; it is a default. A startup path, an environment variable, or an allow_live=true flag flips delivery on before anyone meant to activate it.

The safe posture is the inverse: external delivery should be unavailable unless someone deliberately opened it. Queuing an event for later is harmless and useful; delivering it is the act that needs an explicit, recorded gate. Most webhook layers ship the other way around — wired to deliver, with dormancy as an afterthought you have to remember to configure.

Webhook & Event Dormancy Manager makes dormancy the default and activation the exception. Events are queued durably and can be inspected, but every delivery, replay, autodispatch, and outbound integration is checked against a runtime policy that stays locked unless a specific approval and a well-formed activation mode are both present. If the configuration is malformed in any way, the manager fails closed.

How It Works

Queue → Evaluate → Activate or block

1
Queue, don't deliver

queue_webhook_event writes a durable queued event to the local event log against an active destination. The contract is explicit: queue only — no HTTP send, no retry, no source mutation. Queuing an event never performs an outbound call; it parks the intent for a later, gated decision.

→
2
Evaluate the dormancy policy on every external action

Before any delivery, replay, or Slack send, evaluate_webhook_runtime_policy is consulted. The webhook is webhook_locked by default; it unlocks only when the dormancy lock is off, WEBHOOK_ACTIVATION_APPROVAL is approved, the activation mode is valid, and no routing flags are malformed. Every *_allowed flag is AND-gated on all of these.

→
3
Activate explicitly, or stay blocked with a reason

A blocked action returns a blocked_dormancy_detail string naming exactly why (e.g. "webhook dormancy lock active"). Live delivery via deliver_webhook_event, autodispatch, dead-letter replay_dead_letter_event, and Slack inbound/outbound each require their own flag and an unlocked policy. Nothing escalates from a queued event to an external call without that branch.

Webhook & Event Dormancy Manager is a decision-support and safety layer. It enforces a dormant-by-default posture and surfaces exactly what is and is not activated; it does not decide on your behalf whether an external integration should be turned on, and it does not replace production access controls. Queuing never sends — only an explicit, approved activation does.
Dormancy Model

Modes, gated capabilities & fail-closed

The model below is derived directly from the originating agent's webhook runtime policy. An activation mode and a set of routing flags decide what — if anything — is permitted to leave the machine. Absence or garbage means locked; activation requires positive, well-formed configuration.

Activation modes
ModeMeaning
dormant Default. No external delivery of any kind. Unknown or malformed modes collapse to this.
manual_dispatch Per-event delivery permitted only when policy is unlocked and WEBHOOK_AUTODISPATCH_ALLOW_LIVE is set.
autodispatch Automatic delivery permitted only when unlocked and WEBHOOK_AUTODISPATCH_ENABLED is set.
Gated capabilities

Every capability is AND-gated on the policy being unlocked — lock off and approval present and mode valid and no malformed flags — plus the additional requirement below.

CapabilityAdditional requirement (on top of unlocked)
live_delivery_allowed WEBHOOK_AUTODISPATCH_ALLOW_LIVE true, mode ∈ {manual_dispatch, autodispatch}
autodispatch_allowed mode = autodispatch, WEBHOOK_AUTODISPATCH_ENABLED true
manual_dispatch_allowed mode = manual_dispatch
dead_letter_replay_allowed WEBHOOK_DEAD_LETTER_REPLAY_ENABLED true, mode ∈ {manual_dispatch, autodispatch}
slack_outbound_queue_allowed SLACK_OUTBOUND_ENABLED true
slack_inbound_delivery_allowed SLACK_INBOUND_DELIVERY_ENABLED true
Fail-closed rule

A malformed WEBHOOK_DORMANCY_LOCK, an unrecognized activation mode, or any malformed routing flag forces dormancy and is named in the policy reason. Activation requires positive, well-formed configuration — absence or garbage means locked.

evaluate_webhook_runtime_policy never opens on uncertainty: every *_allowed flag is AND-gated on lock-off, an approved WEBHOOK_ACTIVATION_APPROVAL, a valid mode, and the absence of malformed flags. Queuing an event never sends — only an explicit, approved, well-formed activation branch can permit an external call.

Lineage note. The originating agent's stance is fail-closed dormancy — webhook delivery, Slack send, autodispatch, retries, and replay stay unavailable unless an explicit gated activation branch is opened. The product packages that exact posture; the only generalization is widening the originating agent's fixed event types and Slack-specific routing into a configurable surface (proposed). Source: the originating agent's webhook runtime + external-routing dormancy.
MCP Interface

MCP tools

The proposed MCP surface below is derived from what the originating agent's webhook routes and services expose today. This is the interface of the built Wave 2 package.

Queue & Inspect
queue_webhook_event
Queue a durable outbound event (queued, no send) against an active destination.
list_webhook_event_log
List queued / delivered / failed events from the local event log.
list_webhook_destinations
List configured destinations and their allowed event types.
list_dead_letters
List events parked in the dead-letter state.
Policy & Dormancy
get_dormancy_policy
Return the evaluated runtime policy: lock state, activation mode, and every *_allowed flag with a reason.
get_external_routing_summary
Admin summary of destinations, recent events, and the current activation posture.
Gated External Actions
deliver_webhook_event
Attempt delivery of one queued event; blocked unless policy permits (allow_live requires live_delivery_allowed).
replay_dead_letter_event
Replay a dead-lettered event; blocked unless dead_letter_replay_allowed.
process_webhook_queue
Process the queue; gated by the dormancy policy on every external action.

Scope

What Webhook & Event Dormancy Manager does and does not do

Does
  • ✓ Stay dormant by default — no external delivery without explicit activation
  • ✓ Queue events durably so they are not lost while dormant
  • ✓ Fail closed on a malformed lock, mode, or routing flag — garbage means locked
  • ✓ Gate live delivery, autodispatch, replay, and Slack independently and explicitly
  • ✓ Surface a human-readable reason for every blocked action
  • ✓ Run fully local as an MCP server, SDK, and local HTTP API
Does Not
  • – Send external calls while dormant — no delivery while the dormancy lock is active
  • – Activate without explicit approval, or treat missing/garbage config as permission
  • – Send during queuing — queuing never performs an HTTP call
  • – Decide for you whether an integration should be on
  • – Guarantee delivery, ordering, or third-party acceptance once activated
  • – Replace production webhook infrastructure or its access controls
