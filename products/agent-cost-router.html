The Problem

Every task goes to the same expensive model.

Agent workloads are not uniform. A prompt that asks "what is the capital of France?" does not need the same model as one that asks to redesign a production authentication architecture. But most agent pipelines route everything to a single model — usually the most capable one available — regardless of what the task actually requires.

The result is unnecessary spend, slower latency on tasks that a cheaper model would handle equally well, and no record of what each routing decision cost or why it was made. Teams have no visibility into how much agent work actually costs, which tasks consumed the budget, or whether the model choice was justified.

Agent Cost Router adds a classification and routing layer before each prompt reaches a provider. Simple tasks go to nano-tier models. Complex reasoning goes to the tier it needs. Every decision is logged with cost estimate, savings, and alternatives considered.

How It Works

Classify → Route → Log

1
Classify prompt complexity

The classifier reads the prompt and assigns a complexity tier — nano, standard, or advanced — based on length, token estimate, reasoning signals, and caller-supplied constraints like requiresTools or minContextWindow.

→
2
Select the cheapest capable model

The router filters the model catalog by tier, provider constraints, tool and vision support, context window, and budget cap. The selected budget_mode — cheapest, balanced, or performance — controls how the winner is chosen from the filtered set.

→
3
Execute and log the decision

Dry-run mode returns the routing decision and cost estimate without calling a provider. Run mode executes the prompt, records actual tokens and cost, and appends the result to the persistent local history with savings vs. the claude-sonnet baseline.

Agent Cost Router is a local routing and cost-accounting layer. Model pricing is only as fresh as the static catalog or the last OpenRouter sync. Provider execution still requires the relevant API key configured in the environment.

Routing Policy

Three budget modes. One consistent interface.

Every route call takes a budget_mode. The classifier determines the minimum capable tier. The budget mode controls how the winner is selected from that tier and above.

Cheapest

Picks the lowest-cost model that meets the minimum tier requirement. Free models always win over paid; among free models, the one with the largest context window is preferred. Best for high-volume workloads where cost minimization is the primary goal.

Balanced

Prefers the minimum capable tier at lowest cost, but applies a mild penalty for over-provisioning to a higher tier unnecessarily. A sensible default for mixed workloads where some tasks need room to grow.

Performance

Picks the highest-tier model available before considering cost. Use this for critical reasoning tasks, production analysis, or any work where capability matters more than spend.

Local Install

CLI, MCP server, and local dashboard

Agent Cost Router ships as a local CLI tool and MCP server. Once installed, use the CLI for dry-run routing and one-off prompts, or wire the MCP tools into any MCP-compatible agent runtime.

The local dashboard starts on http://127.0.0.1:4317/dashboard after running agent-cost-router serve.

View on npm →
npm package 
# Install locally for MCP config
npm install @certaworks/agent-cost-router

# Optional: install globally for direct CLI commands
npm install -g @certaworks/agent-cost-router
CLI usage 
# Dry-run: classify and pick model, no execution
agent-cost-router route \
  --prompt "Summarize this paragraph briefly"

# Run: route and execute, log result
agent-cost-router run \
  --prompt "Analyze this architecture design" \
  --provider anthropic \
  --budget-mode balanced

# View cost history and savings ledger
agent-cost-router history list --project demo
agent-cost-router budget summary \
  --project demo --budget-id q2

# Start local dashboard + HTTP API
agent-cost-router serve --port 4317

MCP Interface

Available MCP tools

Agent Cost Router exposes the following tools to any MCP-compatible agent runtime.

Routing
route_prompt
Classify a prompt and pick the cheapest capable model — dry run only. Returns model, tier, estimated cost, and savings vs. claude-sonnet.
run_prompt
Route a prompt to the cheapest capable model and execute it. Returns the model response plus a full cost breakdown.
Model Catalog
list_models
List known models sorted by cost. Supports filtering by tier, provider, and free-only. Includes both static and live-synced models.
sync_models
Fetch the latest model list from OpenRouter and update the local cache. Requires OPENROUTER_API_KEY.
estimate_cost
Calculate the estimated cost in USD for a specific model and token counts without routing or executing.
History & Budget
list_route_history
List persisted routing and run history records, newest first. Filter by project or budget ID.
get_budget_summary
Summarize estimated cost, actual cost, and total savings by project or budget ID across all recorded history.
Dashboard & History

Local dashboard — prototype status

Agent Cost Router includes a local HTTP API and HTML dashboard for reviewing routing decisions, model selections, cost history, and savings ledger. The dashboard starts alongside the local server and requires no cloud connection.

Route history is persisted to ~/.agent-cost-router-history.json by default. The store records sanitized prompt previews, SHA-256 prompt hashes, model, tier, complexity, estimated and actual cost, savings, and alternatives considered. Full prompts are never stored.

Roadmap — not currently built

Hosted SaaS dashboard, team accounts, persistent cloud history, managed budget alerts, and shared savings ledgers are roadmap items. They are not currently built or available. What is available today is the local CLI, MCP server, local HTTP API, and local prototype dashboard.

Scope

What Agent Cost Router does and does not do

Does
  • ✓ Classify prompt complexity into nano, standard, or advanced tier
  • ✓ Route to the cheapest capable model given provider, budget, and constraint filters
  • ✓ Estimate and record cost, savings, and alternatives considered per routing decision
  • ✓ Expose CLI, MCP tools, and local HTTP API for integration into agent pipelines
  • ✓ Sync live model pricing from OpenRouter when an API key is present
Does Not
  • – Guarantee pricing accuracy — static catalog and OpenRouter sync may lag provider changes
  • – Execute prompts without a valid API key for the selected provider
  • – Provide hosted SaaS, team accounts, or managed cloud history (roadmap)
  • – Evaluate action safety or replace Confidence Gate for pre-action review
  • – Intercept or route requests made directly through provider SDKs
