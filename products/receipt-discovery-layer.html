Receipt Discovery Layer

Proof-of-action gets created everywhere — tool calls, model usage, file reads, memory writes, change logs — across many subsystems and sessions. Receipt Discovery is the one read-only place to ask “did this actually happen?” and get back the matching receipts with a trust classification, not a guess.

The Problem

Receipts pile up in silos. The question gets answered from narration.

Proof-of-action lands wherever it is made. A model call lands in model_usage_log, a tool result in tool_audit, a file read in shared_files, a memory write in a memory table, a state change in change_log — each subsystem keeps its own proof, in its own shape, scoped to its own session. When someone later asks “did the agent actually do X?”, the proof exists, but it is scattered across a dozen surfaces with no shared query.

So the question gets answered from narration instead of receipts. The agent reasons about whether something happened, and confident language fills the gap the missing query left open. Worse, two true things get conflated: a real internal method — a repeatable protocol with traces across memory, execution, and change surfaces — gets described as an external worker or agent run, even though no worker or model receipt exists. The “drone” lesson in the originating agent was exactly this: the method was real and receipt-backed, but it was not an autonomous agent.

Receipt Discovery makes the whole proof-trail queryable. One read-only search fans out across every receipt surface, correlates the hits, and returns them with an explicit trust classification and language guidance — so claims can be bound to receipts instead of to confidence.

How It Works

Search → Classify → Correlate

1
Search every surface

A caller invokes search_receipts with a query, an optional claim_type (unknown / external_action / internal_method), and an optional current session ID. The single real route is POST /api/receipts/search. The layer fans out across every registered trace surface, term-matching the searchable text columns of each — read-only, mutating nothing.

→
2
Classify into a trust taxonomy

Hits are grouped by surface and group, then classified: a direct external receipt → CONFIRMED; a strong cross-surface internal pattern → CONFIRMED as an internal method (explicitly not an external worker); multi-surface but weaker → PARTIAL_EVIDENCE; a single surface → a weak trace; nothing → ABSENT.

→
3
Correlate & guide

Results carry counts by surface, by group, and by session scope (same_session / cross_session / unknown_session / no_session), plus language_guidance and recognition_guidance telling the caller exactly what the receipts do and do not support.

Receipt Discovery is a decision-support layer. It reports what receipts exist and how strong they are — it does not decide whether a claim is “true enough” to act on; the human and the agent do. It searches local surfaces only, and reports honestly which surfaces it searched and which it omitted.

External-receipt groups vs. internal-method groups are tracked separately (EXTERNAL_RECEIPT_GROUPS, INTERNAL_METHOD_GROUPS) — this is what keeps a real internal method from being reported as an external worker.

Trust taxonomy (classification output)
Statusevidence_kindWhen
CONFIRMED external_receipt claim_type=external_action and a direct external-group receipt exists.
CONFIRMED internal_method ≥4 surfaces incl. memory and an execution/change/witness/stream/retraction trace — confirmed as a method, explicitly not an external worker.
PARTIAL_EVIDENCE cross_surface_trace Traces on ≥2 surfaces but not enough for the stronger claim.
PARTIAL_EVIDENCE single_surface_trace A related trace on only one surface — a weak trace, not confirmation.
ABSENT absent No matching traces; treat as absent or hypothesis only.

The full status vocabulary is CONFIRMED / PARTIAL_EVIDENCE / INFERRED / ABSENT. The decisive line is external receipt vs. internal-method trace — a real method is never reported as an external worker run.

Discovery / correlation query dimensions
DimensionField(s)Use
Query terms query (≤8 terms) Free-text match across the searchable columns of each surface.
Claim type claim_type Steers external-action vs. internal-method classification.
Surface / group counts_by_surface, counts_by_group See where the proof lives and how it is distributed.
Session scope counts_by_scope, per-hit session_scope Separate current-session facts (same_session) from cross_session / unknown_session / no_session evidence.
Coverage searched_surfaces, omitted_surfaces Know exactly what was searched and what errored or was skipped.

recognition_guidance turns the session-scope counts into explicit language rules — e.g. cross-session-only evidence may be cited but must not be claimed as current-session direct recognition.

The single read-only route (POST /api/receipts/search), the full surface registry, the trust taxonomy, and the external-vs-internal-method distinction are all implemented and verifier-passed in the originating agent — RECEIPT_DISCOVERY_LAYER_VERIFIER_OK. MCP tools

Discovery
search_receipts
Read-only fan-out search across all local receipt surfaces; returns the matching evidence, a trust classification (CONFIRMED / PARTIAL_EVIDENCE / INFERRED / ABSENT), counts by surface/group/scope, and language + recognition guidance. Wraps POST /api/receipts/search.
Companion tools · proposed — not yet distinct routes in source

Scope

What Receipt Discovery does and does not do

Does
  • ✓ Search every local receipt surface from one read-only query
  • ✓ Classify results into a confirmed / partial / inferred / absent taxonomy
  • ✓ Distinguish a real internal method from an external worker/agent run
  • ✓ Correlate hits by surface, group, and session scope
  • ✓ Emit explicit language + recognition guidance per result
  • ✓ Run fully local as an MCP server, SDK, and local HTTP API
Does Not
  • – Create, write, or modify any receipt (read-only)
  • – Decide whether a claim is “true enough” to act on
  • – Assert an external action without a direct external receipt
  • – Reach across machines or non-local stores — it searches local surfaces only
  • – Guarantee completeness — it reports searched and omitted surfaces honestly
  • – Provide hosted storage, team accounts, or replace human/audit review
