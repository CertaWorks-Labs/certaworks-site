Curiosity Protocol Controller

Letting an agent decide what to explore next is useful and dangerous, so the brakes are built in first. The controller lets an agent propose a bounded, receipt-backed curiosity thread — and nothing more — until a human approves it. Every proposal passes an activation gate, an enforcement matrix of code-level caps, lifecycle stop controls, and human-held pause/stop tokens. Noticing is not acting: autonomy here is opt-in, gated, capped, and human-stoppable by design.

The Problem

The version of autonomy that goes wrong quietly.

The appealing version of agent autonomy — "let it notice interesting threads and chase them" — is exactly the version that goes wrong quietly. An agent that can self-initiate can wander off-task, fabricate a justification for a thread it "felt curious" about, invoke tools nobody approved, write memory, or kick off background work that no human asked for and no human is watching.

Most autonomy designs bolt safety on afterward. The result is a system that already runs and then gets a stop button. The Curiosity Protocol Controller inverts that order: the safety machinery — the receipt requirement, the activation gate, the enforcement matrix, the caps, and the stop controls — is the product. Pursuit is the part that stays off by default.

So the controller draws a hard line between noticing and acting. An agent may submit an explicit proposal with an accepted receipt and exactly one bounded question. The controller classifies it (display-only, blocked, or needs-human-approval), holds it in a queue, and refuses to pursue it. Any step beyond display — even a local read-only review — requires a prior human approval stamp and stays inside enforced caps. This is the highest-risk tool in the suite to describe, so it is described honestly: controlled autonomy with the brakes wired in first.

How It Works

Propose → Gate → Act only within caps

1
Propose, receipt-first — never pursue

An agent calls submit_proposal with an initial_observation_receipt (an accepted form — message id, route output, memo path, fixture id, witness report ref, visibility ref, or explicit user statement) and exactly one bounded question. classify_proposal marks it PENDING_DISPLAY_ONLY, BLOCKED_NO_RECEIPT, BLOCKED_SCOPE_TOO_WIDE, or NEEDS_OPERATOR_APPROVAL. A vague "I felt curious" with no receipt is rejected — and nothing is pursued.

→
2
Gate before any autonomy — human stamps

Before anything beyond display, the controller consults its activation gate, readiness gate, enforcement matrix, and trial caps — and (in Wave 2) the Confidence Gate and Agentic Regulation Engine. A human applies a review_proposal stamp: dismiss, keep_display_only, approve_local_read_only_review, or escalate_to_triad. No stamp, no step.

→
3
Act only within caps, only after approval

run_local_review runs only with a prior APPROVED_LOCAL_READ_ONLY_REVIEW stamp, reads only allowlisted local sources (memos, fixtures, supplied receipt summaries), and emits a receipt to the audit ledger / Witness. Human pause/stop tokens (OPERATOR_PAUSE_CURIOSITY_TRIAL, OPERATOR_STOP_CURIOSITY_TRIAL) halt the trial at any time.

The controller never pursues a thread autonomously, never runs in the background, and every step beyond a display-only proposal requires an explicit human approval recorded as a receipt. It is a gated, decision-support layer — it surfaces proposals and enforces caps so a human can decide; it does not replace human supervision or production controls.
Safety Controls

Activation gate, enforcement matrix & stop controls

The model below is derived directly from the originating agent's curiosity activation gate, enforcement matrix, and lifecycle stop-control memos. The safety machinery is the product; pursuit stays off by default.

Autonomous pursuit is NOT live — it sits behind human approval

For the originating agent, full autonomous curiosity pursuit is not live. What exists today is exactly the gated surface: explicit proposal intake, manual human review stamps, approved local read-only review against allowlisted sources, and durable audit readback. Anything beyond a local read-only review is proposed and always remains behind an explicit human approval gate.

This is not "an autonomous agent that does whatever it wants." It is controlled autonomy: opt-in, receipt-gated, code-capped, and human-stoppable. A human can pause, stop, or signal purge intent at any moment, and there is no background loop that acts without a recorded approval.

Scope

What the Curiosity Protocol Controller does and does not do

Does
  • ✓ Accept explicit, receipt-backed curiosity proposals with exactly one bounded question
  • ✓ Classify each proposal and hold it as display-only until a human approves a next step
  • ✓ Enforce hard caps (3 proposals / 3 local reviews / 1 Triad preflight per trial) and allowlisted sources in code
  • ✓ Run a local read-only review only after an explicit human approval stamp, on allowlisted sources only
  • ✓ Expose readiness, activation, enforcement-matrix, lifecycle, and closeout visibility so safety state is auditable
  • ✓ Honor human-held pause and stop tokens at any time
Does Not — autonomy is gated, opt-in & human-stoppable
  • – Act autonomously without human approval — every step beyond display is human-gated and receipt-logged
  • – Run unsupervised — never runs in the background, schedules work, uses a heartbeat, or "checks later"
  • – Pursue a thread autonomously — autonomous pursuit is not live; it is proposed and remains behind explicit human approval
  • – Infer proposals from live conversation without an explicit submission
  • – Perform web / external research, provider consults, or activate write-capable tools without separate explicit approval; write memory, update goals, revise beliefs, or dispatch webhooks
  • – Act as a loophole for autonomous action, or treat protocol completion as proof of a curiosity signal or "internal experience"
