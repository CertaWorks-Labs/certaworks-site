Contradiction Detector

The Problem

Conflicts don't announce themselves. They compound.

Long-running agents accumulate state: beliefs written to memory, goals set early, statements made mid-session. Over time these drift into conflict — a goal that contradicts an earlier constraint, a belief asserted now that denies one stored before. Nothing in a typical runtime notices. Each new conflicting statement compounds the last, and the agent keeps reasoning on top of an inconsistent foundation.

By the time the inconsistency surfaces in a visibly wrong answer, the contradiction may be many turns old and hard to trace. There is usually no single place that holds the set of conflicts, no record of which were reviewed, and no way to mark one as genuinely resolved versus deliberately dismissed.

Contradiction Detector adds that place. It maintains a durable local map of detected contradictions and gives each one an explicit lifecycle — open until a human or the agent resolves or dismisses it — so conflicts are tracked and closed instead of silently accumulating.

How It Works

Record → Review → Resolve

1
Conflicts are recorded into the map

Detected contradictions are stored as rows in the local contradictions table with a status. In the originating agent the detection upstream feeds this store; the productized tool exposes a record_contradiction write so any detector — yours or a Wave 1 source — can add a conflicting pair to the map with status open.

→
2
The agent reviews the map

Your agent or operator calls list_contradictions, optionally filtered by status and limit, to see open conflicts newest-first (GET /). Each entry carries the conflicting pair and metadata for review.

→
3
Each contradiction is resolved or dismissed

Call resolve_contradiction to mark a reviewed conflict resolved (PATCH /{id}/resolve), or dismiss_contradiction to mark it dismissed (PATCH /{id}/dismiss). The status change is durable, so the map reflects what has been handled.

Contradiction Detector is a decision-support layer. It surfaces and tracks conflicts; it does not decide which side is correct or auto-edit the agent's beliefs. There is no auto-resolution — the tool never picks a winning side, and it does not guarantee it finds every conflict in an agent's state.
Contradiction Model

Map, status lifecycle & review

Grounded in the originating agent's contradictions.py route, which serves the contradiction map and exposes status transitions. The route stores each contradiction as a row with a status and created_at, and supports filtering by status. The status lifecycle below is exact to source; the type taxonomy and severity that follow are proposed product framing (see the lineage note).

Status lifecycle Exact to source
StatusMeaningTransition
open Detected, not yet reviewed. Default state in the map. initial
resolved Reviewed and reconciled — one side corrected, or the conflict genuinely closed. PATCH /{id}/resolve
dismissed Reviewed and deliberately set aside as not a real conflict (or accepted tension). PATCH /{id}/dismiss
Contradiction types Proposed product framing

The source route is type-agnostic — it stores, lists, filters by status, and transitions. The typing below is layered on at productization and is not implemented in source.

TypeDescription
belief_vs_belief Two stored beliefs that cannot both hold.
goal_vs_constraint A goal that conflicts with an earlier constraint or guardrail.
statement_vs_memory A current statement that denies something stored in memory.
identity_inconsistency A claim that conflicts with the agent's recorded identity/state.
Severity Proposed — not shipped
FieldStatus
severity A low / medium / high field proposed at productization to prioritize the map. The originating agent's route does not currently store severity — treat it as a planned field, not a shipped one.

Resolution paths: resolve (reconcile and close), dismiss (set aside), or leave open for human review. There is no auto-resolution — the tool never picks a winning side.

Lineage note. The map, status filtering, and the resolve / dismiss lifecycle are exactly what the originating agent's contradictions.py route ships today. The contradiction-type taxonomy and the severity field are proposed product framing layered on a deliberately simple, type-agnostic store — they are not yet implemented in source.
MCP Interface

MCP tools

The proposed MCP surface below is derived directly from what the originating agent's contradictions.py route exposes today. This is the interface of the built Wave 2 package.

Map
list_contradictions
List contradictions newest-first, optionally filtered by status (open / resolved / dismissed) and limit (GET /).
record_contradiction
Add a detected conflicting pair to the local map with status open — a productized write over the store.
Lifecycle
resolve_contradiction
Mark a contradiction resolved by id (PATCH /{id}/resolve).
dismiss_contradiction
Mark a contradiction dismissed by id (PATCH /{id}/dismiss).

Scope

What Contradiction Detector does and does not do

Does
  • ✓ Maintain a durable local map of detected contradictions
  • ✓ List conflicts newest-first with status and limit filtering
  • ✓ Track each contradiction through open / resolved / dismissed
  • ✓ Provide SDK, CLI, and MCP server surfaces
  • ✓ Keep all state local
Does Not
  • – Decide which side of a conflict is correct
  • – Auto-edit, delete, or overwrite the agent's beliefs or goals
  • – Guarantee it finds every conflict in an agent's state
  • – Replace human review of the agent's reasoning
  • – Provide hosted storage, team accounts, or managed alerting (roadmap)
